# Документация: `app/engine/analyzers`

Этот раздел описывает аналитические модули, которые обогащают натальную карту аналитическими блоками (доминанты, фигуры аспекты, статус планет и компенсаторика).

Файлы:
- `dominants.py` — `calculate_dominants(planets_list, houses_list, aspects_list)`
- `jones_patterns.py` — `calculate_jones_pattern(planets_data)`
- `aspect_patterns.py` — `calculate_aspect_patterns(aspects_list)`
- `planet_status.py` — `calculate_planet_status(aspects_list)`
- `scoring.py` — `get_compensatory_data(planets, aspects)`

---

## `dominants.py`

Описание:
- Вычисляет относительную силу (dominance) десяти классических планет по набору правил, похожих на Astro-Seek.

Входы:
- `planets_list` — список объектов планет (каждый с `name`, `sign`, `house`, `abs_pos`, ...)
- `houses_list` — список домов (каждый с `house`, `sign`, `abs_pos`)
- `aspects_list` — список аспектов (каждый с `p1`, `p2`, `type`, `orb`)

Выход:
- Список объектов `{ name, signPts, housePts, anglePts, rulePts, aspectPts, sum, percent }`, отсортированных по сумме.

Ключевая логика:
- `signPts`: матрица достоинств по знакам (0..6) для каждой планеты.
- `housePts`: бонусы за нахождение в 1/10 доме и в символических домах.
- `anglePts`: большие бонусы за соединение с ASC/MC (если conjunction с малой орбой).
- `rulePts`: подсчёт управлений (кто управляет домами/асцендентом) — даёт дополнительные очки.
- `aspectPts`: очки за мажорные аспекты к другим классическим планетам.

Примечания:
- Итоговые проценты рассчитываются от суммы всех очков в карте.

## `jones_patterns.py`

Описание:
- Анализирует распределение планет по зодиаку и определяет фигуры типа 'bucket', 'seesaw', 'bundle', 'bowl', 'locomotive', 'splay' (классические фигуры по Джонсу).

Входы:
- `planets_data` — список планет с `name` и `abs_pos`.

Выход:
- `None` если меньше 10 планет, иначе объект `{ id, name, focus_planet, empty_start, empty_end }`.

Ключевая логика:
- Сортировка 10 классических планет по абсолютной долготе, вычисление трёх наибольших пустот (gaps), определение фигуры по порогам.

---

## `aspect_patterns.py`

Описание:
- Ищет замкнутые конфигурации аспектов (3-планетные и 4-планетные фигуры): Тау-квадрат, Большой тригон, Йод, Большой крест, Мистический прямоугольник, Парус и др.

Входы:
- `aspects_list` — список аспектов (каждый с `p1`, `p2`, `type`)

Выход:
- Список найденных фигур, каждая с `{ name, type, planets, apex, desc }`.

Ключевая логика:
- Построение быстрого lookup-словаря `aspect_map[p1][p2] = type`, затем перебор комбинаций по 3 и 4 планеты и проверка паттернов по количеству нужных типов аспектов.

## `planet_status.py`

Описание:
- Для каждого объекта карты вычисляет "harmony" и "tension" баллы, а затем проценты гармонии/напряжения.

Входы:
- `aspects_list` — список аспектов

Выход:
- `status_data` — словарь по объектам: `{ "harmony": int, "tension": int, "total": int, "harmony_pct": int, "tension_pct": int }`.

Правила начисления:
- `trine` = +3 harmony, `sextile` = +2 harmony
- `square`/`opposition` = +3 tension
- `conjunction` — зависит от второго участника: с Солнцем в малом орбе добавляет tension (combust), с malefics добавляет tension, с benefics добавляет harmony, иначе +1 harmony.

---

## `scoring.py`

Описание:
- Служит для выделения ключевых точек компенсаторики: угловые планеты, самый точный напряжённый аспект (главная проблема), самый точный гармоничный аспект (главная опора).

Входы:
- `planets` — список планет
- `aspects` — список аспектов

Выход:
- Объект `{ angular_planets, top_challenge_aspect, top_support_aspect }`.

Ключевая логика:
- `angular_planets`: фильтр по домам {1,4,7,10}
- `top_challenge_aspect`: первый по точности (наименьший orb) аспект из типа `square`/`opposition`
- `top_support_aspect`: первый по точности аспект из `trine`/`sextile`

---

Примечания и рекомендации
- Большая часть логики опирается на предположение, что `aspects_list` содержит корректные типы аспектов в нижнем регистре (`trine`, `square`, `sextile`, `conjunction`, `opposition`, `quincunx`).
- Для юнит-тестирования удобно подготовить фикстуры: `planets_list` с несколькими заранее известными конфигурациями и проверять ожидаемые выходы аналитиков.
- Некоторые функции возвращают `None` или пустые структуры в случае нехватки данных; фронтенд должен это корректно обрабатывать.
